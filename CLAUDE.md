# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Build, lint, test

```bash
cargo build --release                                    # binary -> target/release/mcp-server-git-rs
cargo fmt --all -- --check                               # CI gate
cargo clippy --all-targets --locked -- -D warnings       # CI gate (warnings = error)
cargo test --locked --all-targets                        # CI gate
```

Run a single integration-test file: `cargo test --test features` (one file per group under `tests/`). Run a single test: `cargo test --test features -- repo_scoping_blocks_outside_path --exact --nocapture`.

Run the server locally for manual MCP-over-stdio testing:

```bash
cargo run -- --features all -vv -r /path/to/some/repo
```

Benchmark against the Python reference server (requires a release build + `uvx`):

```bash
cargo build --release
cd bench && python3 run.py --out results/$(date +%Y-%m-%d).md
```

## Architecture

### What this is

A Rust MCP server speaking JSON-RPC over stdio, designed as a drop-in replacement for the Python `mcp-server-git` reference. **Tool names, parameter names, default values, and error shapes must match the Python server one-to-one** — that parity is the product. Anything that breaks it is a regression even if it looks like an improvement.

### Crate layout

`src/main.rs` is a thin clap entry point. The library at `src/lib.rs` exposes:

- `server.rs` — the `GitServer` handler. **One giant `#[tool_router]` impl** declares every MCP tool with its `#[tool(...)]` attribute, JSON-Schema-derived param struct, annotations (`read_only_hint`, `destructive_hint`, …), and feature-gate check. The custom `ServerHandler` impl overrides `list_tools` / `get_tool` so feature-gated tools are *hidden* from `tools/list` when their group is disabled; `call_tool` is auto-generated by `#[tool_handler]` and the per-tool `require_feature` early-return handles "called a hidden tool anyway."
- `features.rs` — the `Feature` enum and `FeatureSet` bitmask. CLI parses `--features a,b,c` into the set; `tool_feature(name)` in `server.rs` is the source-of-truth mapping from tool name to gating feature (or `None` for the always-on core 13). Both must stay in sync — there's a test (`tool_router_lists_every_tool`) that asserts the total count is 52.
- `guard.rs` — security-critical. `validate_repo_path` canonicalises input and checks containment against allowed roots; `reject_flag_arg` blocks `-`-prefixed values from reaching libgit2; `require_revparse` is defense-in-depth ref validation. **Every tool that takes user-supplied refs / paths / patterns must run these guards before touching libgit2.**
- One module per feature group: `tools.rs` (core), `inspection.rs`, `tags.rs`, `stash.rs`, `remotes.rs`, `history.rs`, `branches_ext.rs`, `worktrees.rs`, `notes.rs`, plus `log.rs` (timestamp parsing for `git_log`) and `push.rs` (auth chain for `git_push`).

### Adding a tool

1. Pick or create the right module under `src/`. Implement `pub fn tool_fn(repo: &Repository, ...) -> anyhow::Result<String>`.
2. Add a `#[derive(Deserialize, schemars::JsonSchema)]` param struct in `server.rs` (group-grouped — see the existing `// ---------- foo ----------` banners).
3. Add a `#[tool(name = "git_foo", description = "...", annotations(...))]` async wrapper inside the `#[tool_router]` impl. Open the repo via `self.open(&p.repo_path)?` (which runs `validate_repo_path`), call `self.require_feature(Feature::X)?` if gated, return `into_text(...)`.
4. Map the new tool name in `tool_feature(...)` to its `Feature` (or leave it as a core tool).
5. Bump the count in the `tool_router_lists_every_tool` test if you added a tool.
6. Add an integration test under `tests/<group>.rs` using the `Fixture` in `tests/common/mod.rs`.
7. Update README's tool table and CHANGELOG.

### Auth chain (push / fetch / tag-push)

The chain is **ssh-agent → libgit2 credential helper → `git credential fill` → `$MCP_GIT_TOKEN`** for HTTPS. The fallback to shelling out `git credential fill` exists because libgit2's helper handling can't invoke custom helpers like `!gh auth git-credential` or talk to osxkeychain; do not remove it. Push success is detected via libgit2's `push_update_reference` callback so server-side rejections (non-fast-forward, hooks) surface as errors even when the transport says OK.

### Conventions worth preserving

- Output strings are user-facing — they often start with a header line like `"Repository status:\n…"` matching the Python server. Don't reformat without checking parity.
- `git2` is built with `vendored-libgit2` + `vendored-openssl`. Release builds are static; release artifacts (CI) cross-compile via the workflow in `.github/workflows/release.yml`.
- Tests use `tempfile::TempDir` + the `Fixture` helper. Some tests shell out to real `git` (worktree setup, etc.) — that's intentional, libgit2 doesn't cover every flow.
- `git_log` timestamp parsing intentionally does **not** support free-form approxidate (`yesterday`, `last tuesday`). The supported forms are documented in README — do not silently expand the grammar without updating that table.
